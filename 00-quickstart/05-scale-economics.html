05-Scale-Economics

Jun 10, 20263 min read

Use this template to add scale economics reasoning to interview problems. Focus on trade-offs and break-even points, not specific dollar amounts.

Database Scaling

Caching Layer:

  • Below 5K QPS: Database alone sufficient (vertical scaling cheaper)
  • Above 5K QPS: Cache becomes economical (hit rate >70%, saves 3.5K DB QPS)
  • Above 20K QPS: Cache required (DB vertical limit ~25K QPS)

Read Replicas:

  • Below 20K read QPS: Primary + cache sufficient
  • Above 20K read QPS: Add replicas (horizontal scale, each adds 10-20K QPS capacity)
  • Above 100K read QPS: Need 5+ replicas + aggressive caching

Sharding:

  • Below 5TB data: Single DB cluster sufficient
  • Above 5TB data: Sharding improves query latency (keep working set in memory)
  • Above 10TB data: Sharding required (single instance memory limits)

Trade-off Quantification

01-01-Redis vs Memcached:

  • Performance: Memcached 15% faster for simple get/set (less overhead)
  • Memory: Redis uses 1.8x more memory (persistence + data structures)
  • Complexity: Redis +30% ops (backup, AOF/RDB management, replication lag)
  • Features: Redis provides sorted sets (timeline), pub/sub (real-time updates)
  • Decision: Use Redis for timeline storage (need sorted sets + persistence)

Fanout-on-write vs Fanout-on-read:

  • Write latency: Fanout-on-write is 100x slower (write to N follower timelines)
  • Read latency: Fanout-on-write is 10x faster (pre-computed timeline)
  • Storage: Fanout-on-write uses 50x more storage (timeline per user)
  • Celebrity problem: Fanout-on-write fails for 10M+ followers (timeout)
  • Decision: Hybrid - fanout-on-write for regular users (<10K followers), fanout-on-read for celebrities

Synchronous vs Asynchronous Ch05-Replication:

  • Write latency: Sync adds 50ms (RTT to replica DC)
  • Durability: Sync = 0 data loss, Async = up to 1s data loss
  • Throughput: Async is 2x higher (no replication blocking)
  • Decision: Async replication for tweets (1s data loss acceptable), sync for payments

Common Scale Thresholds

Caching

  • 5K QPS: Cache becomes economical if hit rate >70%
  • 20K QPS: Cache layer required
  • 100GB working set: Single Redis instance sufficient
  • 1TB working set: Redis cluster required (multiple nodes)

Message Queue

  • 1K events/sec: Simple queue (SQS, RabbitMQ) sufficient
  • 10K events/sec: Need Kafka or Pulsar (horizontal scaling)
  • 100K events/sec: Kafka with multiple partitions (10-50 partitions)

CDN

  • 1Gbps egress: CDN becomes economical (offload 80-90%)
  • 10Gbps egress: CDN required (origin can’t handle load)
  • Global users: CDN required for <100ms latency

Replication

  • Async replication: Acceptable for most use cases (1s lag tolerable)
  • Sync replication: Required for money/inventory (0 data loss)
  • Multi-region: Required when >30% users outside primary region
